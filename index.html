Deployment & Infra

從 bare metal / GPU 到 Runtime、容器、可觀測性與成本控制。我的目標是讓 AI 系統「能長期穩定運行」。

System layers

  • Compute：GPU/CPU 選型、CUDA/Driver、資源隔離。
  • Runtime：Ollama / vLLM，模型版本、快取、吞吐策略。
  • Data：Qdrant、Embedding pipeline、增量更新與回滾。
  • Ops：Logs/metrics/traces、告警、故障復原（runbook）。

Operational checklist

  • 健康檢查：GPU/Runtime/向量庫/工具端點。
  • 可觀測性：每次回答的延遲、token、命中率、工具錯誤。
  • 成本：快取、批次 embedding、路由與降級策略。
  • 安全：金鑰管理、最小權限、審計 Log。

Integration patterns

  • API Gateway + Function Tools：統一 auth、rate limit、審計。
  • Event-driven：事件進來 → AI 分析 → 建議/摘要 → 通知/回寫。
  • Human-in-the-loop：高風險動作需要人工確認。

Deliverables

  • 架構設計 + 選型理由（Trade-offs）。
  • 部署文件、runbook、告警策略。
  • 可測試與可回放（含審計日誌）。

Security & Privacy

企業最怕的不是 AI 不聰明，而是資料外洩與不可控。我的設計預設「資料邊界、最小權限、審計可追溯」。

Key rules

  • 資料不外流：可選 on-prem LLM；敏感資料避免出網。
  • 最小權限：RBAC、工具白名單、上下文裁剪。
  • 全程審計：提示/引用/工具呼叫/結果都可追溯。
  • 安全護欄：輸入輸出過濾、敏感資訊遮罩、速率限制。

AI Agent Case Study

已上線的 AI Agent：以對話驅動方式維護 Lark 人員資料，將多步驟人工操作轉為可控、可審計的工作流。

Overview

  • What：對話式 AI Agent（Lark User / Identity Management）
  • Scope：建立/刪除帳號、改名、改上級、暫停/啟用
  • Usage：Production in use — 每日約 30–50 次操作
  • Safety：Hard-stop 防呆 — 必要資訊不足即不執行
  • Stack：Lark Bot(Webhook) + LLM(Intent/slot) + Tag Routing + AWS Backend

Architecture

User (Lark Chat)
  ↓
Lark Bot (Webhook event)
  ↓
LLM (Intent parsing + slot extraction)
  ↓
Structured validation (required fields)
  ↓
Tag routing (create / delete / update / suspend / enable)
  ↓
Automation workflow (per action)
  ↓
AWS backend (long-running service)
  ↓
Lark Open API (user management)

除了聊天功能，也把「意圖 → 欄位 → 流程」做成可控、可維護、可追溯的系統。

Intent & slot design

LLM 負責意圖分類與欄位抽取（slot filling）並輸出結構化資料。現階段採「先擋風險」策略： 必要欄位不足時直接拒絕執行（hard-stop）。下一步規劃加入 feedback 引導補欄位（slot-check + re-prompt）。

Intent Structured Output Validation

Production stability

AWS 常駐服務承接 webhook 事件；目前已在 production 穩定運行 10+ 天，日常被實際流程使用。

AWS Webhook Service

Security & guardrails

以動作白名單（allowlist）與必要欄位檢查控風險：可解析但資訊不完整時「不執行」。 變更行為封裝在受控的 workflow / automation 中，降低誤操作風險並便於後續審計。
